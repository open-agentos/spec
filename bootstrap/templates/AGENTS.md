## Architecture

GitHub labels drive a state machine across issues and pull requests. Each status
label transition fires the orchestrator workflow (.github/workflows/orchestrator.yml).
The workflow identifies the routing table entry for the new label, mints a
short-lived GitHub App installation token scoped to the repository, injects a
standard set of environment variables, and calls the runner command. The runner
starts the agent process, which reads the issue/PR context, does its work, and
posts a run receipt as a comment before exiting. There is no persistent agent
process — the system is entirely event-driven and stateless between runs.

## Roles

| Role | Triggered by | Can push code? | Notes |
|---|---|---|---|
| builder | status:todo, status:changes-requested | Yes | Power role |
| reviewer | status:in-review | No (intentional) | Cannot push |
| watcher | schedule / settlement | No | Minimal footprint |
| board | PR close | No | Projects v2 only |
| docs | status:approved (if enabled) | Yes | Reuses builder App |
| planner | status:planning (if enabled) | Yes | Reuses builder App |

## State Machine

```
status:planning         -> planner
status:todo             -> builder
status:in-progress      -> (informational)
status:in-review        -> reviewer
status:changes-requested -> builder
status:approved         -> docs (if enabled)
status:blocked          -> HUMAN
status:done             -> terminal
```

## Execution Protocol (Builder)

1. Read issue title, body, and linked PR (if any)
2. Understand what Done looks like before touching any file
3. Make the smallest change that satisfies the issue
4. Run tests / lint if the project has them
5. Open a PR from branch `agent/builder/{issue}-{slug}`
6. Apply `status:in-review` to the issue
7. Exit cleanly (exit 0)

## Execution Protocol (Reviewer)

1. Read the PR diff carefully
2. Check: scope (does it match the issue?), correctness, style
3. If approved: apply `status:approved` to the issue
4. If changes needed: apply `status:changes-requested` + `review:scope-violation` if relevant
5. Post a detailed review comment before changing the label
6. NEVER push commits to the branch

## Guardrail Rules

- No infinite label loops: do not re-apply the same status label you just removed
- If stuck after 2 retries: apply `status:blocked` and explain in a comment
- Max turns: respect AGENT_MAX_TURNS env var; set status:blocked before exiting if incomplete
- Secrets: never log or echo credential values; never commit .env or *.pem files
- Scope: only modify files relevant to the issue; do not refactor unrelated code
